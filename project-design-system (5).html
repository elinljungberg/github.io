Rebuilding a design system for a new brand era

When the company rebranded, we didn't just swap logos. We took the opportunity to rebuild our fragile design system with a new token structure, expanded component library, proper documentation, and full WCAG compliance.

Year

2025

Role

UX Designer

Tools

Figma

Team

Design + Frontend

The problem

The company had a design system, but it was showing its age. Built up over several years, it had accumulated inconsistencies, undocumented workarounds, and a token structure that no longer reflected how the product actually worked. Components varied across different parts of the product, accessibility had never been formally addressed, and the codebase had grown fragile in ways that made even small changes risky.

The catalyst for change came from marketing: a full rebrand was in the works, and it would affect every color, typeface, and visual pattern in the product. Rather than patching the new brand onto the old system, we used the moment as an opportunity to rebuild properly and do it right this time.

The rebrand wasn't just a visual refresh. It was the forcing function we needed to fix what had been silently broken for years.

My role

I led the design side of the rebuild, auditing what existed, defining the new token and variable structure, expanding the component library, and writing the documentation. I worked closely with frontend developers throughout to make sure design decisions translated cleanly into code and that the new system was actually more stable than the one it replaced.

Accessibility was a shared responsibility, but I owned making sure every component and style guide met WCAG, something the old system had ignored.

Process

We started with a thorough audit of the existing system, cataloging every component, token, and style to understand what was worth keeping and what needed to go. The old token structure had grown organically and inconsistently, with raw hex values scattered throughout components rather than a proper semantic layer. That was the first thing to fix.

The new token and variable structure was designed in close collaboration with the frontend team, so that names made sense both in Figma and in code. With the brand change driving new color palettes and typography, we rebuilt the primitive and semantic token layers from scratch, making theming systematic rather than manual.

Alongside the token work, we expanded the component library significantly. Each component was built to meet WCAG AA as a baseline, with contrast ratios, focus states, and keyboard navigation documented as part of the spec rather than treated as afterthoughts. New documentation was written for every component, covering usage, variants, dos and don'ts, and accessibility notes.

Key design decisions

A design system is a series of small, permanent decisions. Here are the ones I'm most proud of.

Rebuilding the token structure from the ground up

The old system used raw values directly in components with no semantic layer, no theming logic. We introduced a two-tier structure separating primitive tokens (raw values) from semantic tokens (like color-action-primary).

WCAG compliance as a design constraint, not a checklist

Accessibility had been treated as something to fix after the fact in the old system. This time, contrast ratios, focus states, and touch targets were defined at the token level, so any component built from the system inherits accessible defaults rather than having to opt in.

Documentation that explains the why

The old system had almost no documentation, which meant every decision lived in someone's head. We wrote usage guidelines for every component, including the reasoning behind decisions, so future designers can make informed exceptions rather than just guessing at intent.
